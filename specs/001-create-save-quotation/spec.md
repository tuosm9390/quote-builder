# Feature Specification: 견적서 생성 및 저장 기능 (Create and save quotation)

**Feature Branch**: `001-create-save-quotation`  
**Created**: 2026-03-17  
**Status**: Draft  
**Input**: User description: "견적서 생성 및 저장 기능"

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 신규 견적서 작성 및 저장 (Priority: P1)

사용자는 새로운 프로젝트를 위한 견적서를 작성하고 시스템에 안전하게 저장할 수 있어야 합니다.

**Why this priority**: 가장 핵심적인 가치 제안이며, 다른 모든 기능(PDF 내보내기, 상태 관리 등)의 토대입니다.

**Independent Test**: 사용자가 견적서 제목을 입력하고 블록을 추가한 뒤 '저장' 버튼을 눌렀을 때, 데이터베이스에 정상적으로 반영되고 목록에서 확인할 수 있어야 합니다.

**Acceptance Scenarios**:

1. **Given** 빈 견적서 작성 화면에서, **When** 제목을 입력하고 텍스트 블록과 가격표 블록을 추가한 뒤 '저장' 버튼을 누르면, **Then** "저장되었습니다" 메시지가 표시되고 견적서 ID가 생성됩니다.
2. **Given** 저장된 견적서가 있는 상태에서, **When** 제목이나 블록 내용을 수정한 뒤 다시 '저장' 버튼을 누르면, **Then** 기존 데이터가 최신 내용으로 업데이트됩니다.

---

### User Story 2 - 가격표 블록을 통한 자동 금액 산출 (Priority: P2)

사용자는 견적서 내부에 가격표 블록을 추가하고, 품목별 단가와 수량을 입력하여 총액이 자동으로 계산되는 것을 볼 수 있어야 합니다.

**Why this priority**: 수동 계산의 번거로움과 오류를 줄여주는 핵심적인 편의 기능입니다.

**Independent Test**: 가격표 블록에 품목을 추가하고 수량을 변경했을 때, 하단의 합계 금액이 실시간으로 업데이트되는지 확인합니다.

**Acceptance Scenarios**:

1. **Given** 가격표 블록이 포함된 견적서에서, **When** 특정 항목의 수량을 '2'에서 '5'로 변경하면, **Then** 해당 항목의 소계와 견적서 전체 총액이 즉시 재계산됩니다.

---

### Edge Cases

- **빈 제목 저장**: 제목을 입력하지 않고 저장을 시도할 경우 사용자에게 알림을 주어야 합니다.
- **네트워크 오류**: 저장 도중 서버와의 연결이 끊겼을 때, **브라우저 로컬 스토리지에 임시 저장**하여 복구 기회를 제공하고 사용자에게 안내해야 합니다.
- **초대형 데이터**: 수백 개의 블록이 포함된 대규모 견적서도 성능 저하 없이 저장 및 로드되어야 합니다. (SC-005 기준)

## Clarifications _(mandatory)_

### Session 2026-03-18
- Q: 견적서에 대한 사용자 권한 모델은 무엇인가요? → A: 소유자 전용 (Owner-only)
- Q: 견적서 상태(Status) 변경은 어떤 방식으로 발생하나요? → A: 수동 변경 (사용자가 UI 버튼을 통해 직접 조작)
- Q: 견적서 삭제 시 물리적으로 삭제하나요, 아니면 상태만 변경하나요? → A: 상태 변경 (Soft Delete)
- Q: 통화(Currency) 및 소수점 처리 방식은 어떻게 되나요? → A: 원화(KRW) 기준, 소수점 없는 정수 처리
- Q: 가격표 블록의 부가세(VAT) 표시 방식은 어떻게 되나요? → A: 선택 가능 (사용자가 부가세 별도/포함 여부를 토글로 선택)
- Q: 동일 사용자가 여러 기기에서 동시 편집 시 충돌 해결 방식은 무엇인가요? → A: 낙관적 잠금 (Optimistic Locking) - 저장 시 버전 체크를 통해 충돌 방지
- Q: 수락(ACCEPTED) 또는 거절(REJECTED) 상태에서 다시 DRAFT로 되돌릴 수 있나요? → A: 불가 (확정된 견적은 수정할 수 없으며 잠금 유지)
- Q: 초대형 데이터(수백 개의 블록) 처리 기준은 무엇인가요? → A: 100개 블록 기준 로딩/저장 1초 이내 완료
- Q: 삭제된 견적서의 사용자 직접 복구(휴지통) 기능이 필요한가요? → A: 미포함 (MVP에서는 데이터 보존만 수행하고 UI 복구 기능은 제외)

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 시스템은 견적서의 고유 ID, 제목, 블록 데이터, 총액, 상태 정보, 버전 번호(`version`), 그리고 삭제 여부(`deletedAt`)를 포함한 견적서 엔티티를 관리해야 합니다.
- **FR-002**: 사용자는 견적서 제목을 필수적으로 입력해야 하며, 텍스트(제목, 본문, 목록 등) 및 가격표 블록을 자유롭게 구성할 수 있어야 합니다.
- **FR-003**: 기본 통화는 원화(KRW)이며, 모든 금액 계산 및 저장 시 소수점은 무시하고 정수형(Integer)으로 처리합니다. 가격표 블록 내의 모든 금액은 단가와 수량을 기반으로 자동 계산되며, 사용자는 '부가세 별도' 또는 '부가세 포함' 여부를 토글로 선택하여 총액에 반영할 수 있어야 합니다.
- **FR-004**: 견적서 저장 시 데이터베이스의 데이터 무결성을 위해 서버 측에서 총액(totalAmount)을 재검증해야 합니다.
- **FR-005**: 모든 견적서 데이터는 JSONB 형식으로 영구 저장되어야 하며, `BlockNote` 라이브러리의 표준 스키마를 따릅니다.
- **FR-006 (상태 관리)**: 견적서는 `DRAFT` -> `SENT` -> `ACCEPTED/REJECTED` 순서로 상태가 전이됩니다. 상태 변경은 사용자가 UI에서 명시적으로 버튼을 클릭할 때 발생합니다. `SENT` 이후 모든 상태(`ACCEPTED`, `REJECTED` 포함)에서는 사용자가 내용을 수정할 수 없도록 잠금 처리해야 하며, 한 번 확정된 상태(`ACCEPTED/REJECTED`)는 다시 `DRAFT`로 되돌릴 수 없습니다.
- **FR-007**: 견적서 데이터는 소유자 본인만 조회 및 수정할 수 있으며, 타인의 데이터에 대한 접근은 RLS(Row Level Security)를 통해 엄격히 금지됩니다.
- **FR-008 (삭제 처리)**: 사용자가 견적서를 삭제할 경우 데이터베이스에서 즉시 제거하지 않고 `deletedAt` 필드에 삭제 일시를 기록하여 목록에서 제외합니다. MVP 단계에서 사용자를 위한 별도의 복구 UI(휴지통)는 제공하지 않습니다.
- **FR-009 (동시성 제어)**: 시스템은 낙관적 잠금(Optimistic Locking)을 구현하여, 클라이언트가 보낸 버전 번호가 서버의 현재 버전과 다를 경우 저장을 거부하고 사용자에게 알림을 주어야 합니다.

### Key Entities

- **Quotation**: 견적서의 기본 정보를 담는 엔티티. 
  - Status: `DRAFT`, `SENT`, `ACCEPTED`, `REJECTED`
  - Fields: `id`, `title`, `blocks`, `totalAmount`, `status`, `version`, `ownerId`, `deletedAt`, `createdAt`, `updatedAt`
- **Block**: 견적서를 구성하는 개별 콘텐츠 단위. 텍스트 유형과 가격표(`pricingTable`) 유형이 존재합니다.

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 사용자는 견적서 작성 시작부터 저장 완료까지 30초 이내에 수행할 수 있어야 합니다. (초기 입력 기준)
- **SC-002 (계산 성능)**: 가격표 블록 내 수치 변경 시 **클라이언트 UI 상의 합계 반영**까지의 지연 시간은 200ms 미만이어야 합니다.
- **SC-003**: 모든 견적서 데이터는 저장 후 페이지를 새로고침했을 때 100% 일관되게 로드되어야 합니다.
- **SC-004**: 서버 측 총액 검증 통과율은 정상적인 요청 시 100%여야 합니다.
- **SC-005 (대규모 데이터 성능)**: 최대 100개의 블록이 포함된 견적서의 로딩 및 저장 작업은 네트워크 환경을 제외한 서버/클라이언트 처리 기준 1초 이내에 완료되어야 합니다.

## Assumptions

- 사용자는 로그인이 된 상태이며, 본인의 견적서에 대해서만 읽기/쓰기 권한을 가집니다. (RLS 정책 적용 전제)
- 견적서 상태의 기본값은 'DRAFT'로 설정됩니다.
